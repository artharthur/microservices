# Домашнее задание: Микросервисы — масштабирование

## Задача 1: Кластеризация

### Предлагаемое решение: Kubernetes

Для оркестрации контейнеров в микросервисной архитектуре я предлагаю Kubernetes — на сегодня это стандарт индустрии, и альтернативы ему по совокупности возможностей нет.

По требованиям:

Поддержка контейнеров — Kubernetes изначально создан для управления контейнерами. Поддерживает Docker, containerd, CRI-O. Каждый микросервис упаковывается в Docker-образ, описывается манифестом (Deployment, Service, Ingress) и деплоится в кластер. Kubernetes следит за тем, чтобы нужное количество реплик работало, перезапускает упавшие поды, раскатывает обновления без даунтайма через rolling update.

Обнаружение сервисов и маршрутизация — встроенный механизм Service Discovery работает из коробки. Каждый сервис при создании получает DNS-имя внутри кластера (например, `cart-service.default.svc.cluster.local`). Другие сервисы обращаются к нему по имени, а kube-proxy или механизм iptables/IPVS маршрутизирует трафик на живые поды. Не нужно руками прописывать адреса — сервисы находят друг друга автоматически.

Горизонтальное масштабирование — увеличение реплик делается одной командой (`kubectl scale`) или изменением поля `replicas` в манифесте. Нагрузка автоматически распределяется между всеми репликами через Service. Нужно больше экземпляров сервиса корзины перед распродажей — поставил 10 реплик вместо 3, и всё.

Автоматическое масштабирование — Horizontal Pod Autoscaler (HPA) масштабирует количество подов на основе метрик: CPU, RAM или кастомных метрик (например, длина очереди). Vertical Pod Autoscaler (VPA) подбирает оптимальные requests/limits для контейнеров. Cluster Autoscaler масштабирует сам кластер — добавляет или убирает ноды в облаке, когда подам не хватает ресурсов или ноды простаивают. Три уровня автоскейлинга, каждый решает свою задачу.

Разделение ресурсов (внешние и внутренние) — Kubernetes чётко разделяет внешний и внутренний трафик. Ingress (или Ingress Controller — тот же Nginx или Traefik) отвечает за трафик извне: принимает HTTP/HTTPS-запросы и маршрутизирует на нужные сервисы. Внутренние сервисы типа ClusterIP доступны только внутри кластера — снаружи до них не добраться. Network Policies позволяют ещё тоньше настроить сетевую изоляцию: например, разрешить сервису платежей общаться только с сервисом заказов и базой данных, а всё остальное запретить. Namespaces дополнительно разграничивают окружения (dev, staging, prod) и команды.

Конфигурация и безопасное хранение секретов — ConfigMaps хранят конфигурацию приложений (переменные среды, конфиг-файлы) и пробрасывают их в контейнеры. Secrets хранят чувствительные данные: пароли, токены, TLS-сертификаты, ключи шифрования. Secrets можно монтировать как файлы или передавать как переменные окружения. Для усиленной безопасности Secrets шифруются at rest через EncryptionConfiguration, а для production-сценариев можно интегрировать внешние хранилища секретов — HashiCorp Vault или cloud-native решения (AWS Secrets Manager, GCP Secret Manager) через External Secrets Operator.

Почему не альтернативы:

Docker Swarm — проще в настройке, но значительно беднее по возможностям. Нет нормального автоскейлинга, слабые network policies, скромная экосистема. Docker фактически сам признал превосходство Kubernetes, добавив его поддержку в Docker Desktop.

Nomad (HashiCorp) — хороший оркестратор, легче Kubernetes, умеет запускать не только контейнеры. Но экосистема меньше, комьюнити меньше, специалистов на рынке найти сложнее. Для крупной компании это существенный фактор.

OpenShift — по сути Kubernetes с обвесом от Red Hat. Отличный продукт, но платный и добавляет свой слой абстракции. Если нет привязки к экосистеме Red Hat, стандартный Kubernetes с open-source дополнениями даёт ту же функциональность.

Kubernetes — это сложно, не буду скрывать. Порог входа высокий, нужны компетентные люди для эксплуатации. Но для крупной компании с микросервисной архитектурой это единственное решение, которое масштабируется от десяти сервисов до тысячи без смены платформы. А managed-варианты (EKS, GKE, AKS) снимают часть операционной нагрузки, если компания работает в облаке.
