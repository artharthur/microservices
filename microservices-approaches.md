# Домашнее задание: Микросервисы — подходы

## Задача 1: Обеспечить разработку

### Предлагаемое решение: GitLab + GitLab CI/CD

Для организации полного цикла разработки в микросервисной архитектуре я предлагаю GitLab — он закрывает все требования одним продуктом, без необходимости интегрировать несколько разрозненных систем.

Как это работает: GitLab выступает одновременно и как система контроля версий (Git), и как CI/CD-платформа. Каждый микросервис живёт в своём репозитории, у каждого — свой `.gitlab-ci.yml` с пайплайном. Всё в одном месте, единый интерфейс, единая авторизация.

Теперь по требованиям:

Облачная система — GitLab доступен как SaaS (gitlab.com), можно не поднимать свою инсталляцию. При этом если потребуется — есть self-hosted вариант, что для крупной компании может быть критично.

Git и репозиторий на каждый сервис — это базовая функциональность GitLab. Группы и подгруппы позволяют удобно организовать десятки репозиториев по командам или доменам.

Запуск сборки по событию — пайплайны стартуют автоматически по push, merge request, тегу. Настраивается через `rules` или `only/except` в конфиге.

Запуск по кнопке с параметрами — есть `when: manual` для ручного запуска и `variables` для передачи параметров. Можно прямо из интерфейса нажать кнопку и указать, например, версию или окружение.

Настройки для каждой сборки — переменные окружения задаются на уровне проекта, группы или конкретного пайплайна. Секреты хранятся в CI/CD Variables с маской и защитой по веткам — пароли и ключи не светятся в логах.

Шаблоны конфигураций — `include` и `extends` позволяют создавать общие шаблоны пайплайнов и переиспользовать их между репозиториями. Создал один шаблон для Go-сервисов, другой для Node.js — и подключаешь в каждом проекте.

Несколько конфигураций из одного репозитория — через `rules` и переменные можно описать разные сценарии сборки в одном `.gitlab-ci.yml`: отдельно для feature-веток, отдельно для main, отдельно для тегов.

Кастомные шаги — каждый `job` в пайплайне это по сути произвольный набор команд. Можно делать что угодно: линтинг, тесты, сборку Docker-образа, деплой, уведомления.

Собственные Docker-образы для сборки — в `image:` указываешь любой Docker-образ, хоть из своего registry. Хочешь собирать в образе с предустановленным Go 1.22 и твоими утилитами — без проблем.

Агенты на собственных серверах — GitLab Runner ставится на любой сервер (Linux, macOS, Windows). Можно поднять раннеры на своём железе, в своём Kubernetes-кластере, с нужными ресурсами и доступами.

Параллельный запуск сборок и тестов — несколько раннеров обрабатывают джобы параллельно. Внутри пайплайна джобы одного stage запускаются одновременно. Для тестов есть `parallel:` — разбивает тесты на части и гоняет их параллельно.

Почему не Jenkins, GitHub Actions или TeamCity: Jenkins мощный, но требует кучу плагинов и постоянного обслуживания — это отдельная головная боль. GitHub Actions хорош, но Git-хостинг и CI/CD живут в разных подсистемах, и для self-hosted варианта он не так удобен. TeamCity — отличный инструмент от JetBrains, но платный для серьёзного использования и менее распространён в DevOps-комьюнити. GitLab же даёт всё из коробки в одном продукте.

## Задача 2: Логи

### Предлагаемое решение: ELK Stack (Elasticsearch + Logstash/Fluentd + Kibana) + Filebeat

Для централизованного сбора логов в микросервисной архитектуре я предлагаю стек ELK — проверенное решение, которое закрывает все требования.

Схема взаимодействия выглядит так: на каждом хосте стоит Filebeat — лёгкий агент, который собирает логи из stdout контейнеров (через Docker/Kubernetes log driver) или из файлов. Filebeat отправляет логи в Logstash (или напрямую в Elasticsearch, если парсинг не нужен). Logstash при необходимости парсит, обогащает и трансформирует записи. Elasticsearch хранит и индексирует всё. Kibana — интерфейс для поиска и визуализации.

По требованиям:

Сбор логов со всех хостов — Filebeat ставится на каждый хост, умеет автоматически обнаруживать контейнеры и собирать их stdout/stderr. Для Kubernetes есть DaemonSet, который разворачивает Filebeat на каждой ноде.

Минимальные требования к приложениям — приложению достаточно писать логи в stdout. Никаких специальных библиотек или форматов не нужно. Filebeat подхватит вывод через стандартный механизм логирования Docker/Kubernetes.

Гарантированная доставка — Filebeat ведёт реестр прочитанных позиций в файлах. Если Elasticsearch недоступен, Filebeat буферизирует данные и дошлёт их, когда связь восстановится. Для дополнительной надёжности можно поставить между Filebeat и Elasticsearch буфер в виде Kafka.

Поиск и фильтрация — Elasticsearch заточен именно под полнотекстовый поиск. Можно искать по любому полю, фильтровать по сервису, уровню логирования, временному диапазону, конкретным ошибкам.

Пользовательский интерфейс — Kibana даёт удобный UI для поиска по логам. Разработчикам создаются отдельные учётные записи с доступом к нужным индексам. Через Spaces в Kibana можно разграничить видимость между командами.

Ссылка на сохранённый поиск — в Kibana можно сохранить любой запрос и поделиться на него ссылкой. Коллега перешёл по ссылке — видит тот же поиск с теми же фильтрами. Очень удобно при разборе инцидентов.

Альтернативой Logstash можно использовать Fluentd (или Fluent Bit) — он легче, потребляет меньше ресурсов и нативно интегрируется с Kubernetes. Для большинства случаев связка Filebeat → Elasticsearch → Kibana достаточна, а Logstash подключается, когда нужен сложный парсинг или трансформация логов.

Из альтернатив всему стеку можно рассмотреть Grafana Loki — он проще и дешевле в эксплуатации, не индексирует содержимое логов (только метки), поэтому потребляет значительно меньше ресурсов. Но полнотекстовый поиск по содержимому у него слабее, чем у Elasticsearch. Для крупной компании, где разработчики активно ищут по логам, ELK остаётся более подходящим выбором.

## Задача 3: Мониторинг

### Предлагаемое решение: Prometheus + Grafana + Node Exporter + cAdvisor

Для мониторинга микросервисной инфраструктуры я предлагаю связку Prometheus + Grafana — де-факто стандарт в индустрии, и не зря.

Схема: Prometheus по pull-модели ходит и собирает метрики с экспортёров и сервисов. Node Exporter стоит на каждом хосте и отдаёт метрики CPU, RAM, диск, сеть. cAdvisor (или встроенные метрики kubelet в Kubernetes) отдаёт метрики по контейнерам — потребление ресурсов каждым сервисом. Сами сервисы отдают свои бизнес-метрики через эндпоинт `/metrics`. Grafana подключается к Prometheus и визуализирует всё это в дашбордах.

По требованиям:

Сбор метрик со всех хостов — Node Exporter разворачивается на каждом хосте (в Kubernetes — как DaemonSet). Prometheus через service discovery автоматически находит все таргеты. Поддерживаются: static config, DNS, Consul, Kubernetes SD и другие механизмы обнаружения.

Метрики хостов (CPU, RAM, HDD, Network) — Node Exporter отдаёт все системные метрики из коробки. Сотни метрик: загрузка процессора, использование памяти, дисковый I/O, сетевой трафик, файловые дескрипторы, и так далее.

Метрики ресурсов каждого сервиса — cAdvisor собирает потребление CPU, RAM, сети и диска на уровне каждого контейнера. В Kubernetes эти метрики доступны через kubelet. Видно, сколько ресурсов ест каждый конкретный сервис.

Специфичные метрики сервисов — Prometheus поддерживает клиентские библиотеки для всех популярных языков (Go, Java, Python, Node.js и т.д.). Разработчик добавляет в свой сервис пару строк кода и начинает отдавать бизнес-метрики: количество заказов в минуту, время обработки платежа, размер очереди — что угодно.

Пользовательский интерфейс с запросами — PromQL (язык запросов Prometheus) позволяет делать выборки, агрегации, вычисления. Например, `rate(http_requests_total{service="cart"}[5m])` — покажет RPS сервиса корзины за последние 5 минут.

Настраиваемые панели — Grafana позволяет создавать дашборды любой сложности: графики, таблицы, heatmaps, алерты. Есть тысячи готовых дашбордов в community (например, Node Exporter Full — готовый дашборд для системных метрик). Каждая команда может настроить свои панели под свои сервисы.

Дополнительно к базовой связке стоит упомянуть Alertmanager — компонент Prometheus для алертинга. Настраиваешь правила (например, «CPU выше 90% дольше 5 минут»), и уведомления приходят в Telegram, Slack, email или PagerDuty. Без алертинга мониторинг — просто красивые графики, а с ним — рабочий инструмент дежурного инженера.

Из альтернатив: Zabbix — мощная система, но заточена под классический мониторинг инфраструктуры и тяжеловата для микросервисов. Victoria Metrics — совместимый с Prometheus storage, имеет смысл подключать как долгосрочное хранилище метрик, когда данных становится много и Prometheus на одном сервере не справляется. Datadog и New Relic — облачные SaaS-решения, работают отлично, но стоят дорого при больших объёмах.
