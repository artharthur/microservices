# Домашнее задание: Микросервисы — принципы

## Задача 1: API Gateway

### Зачем нужен API Gateway

В микросервисной архитектуре клиент не должен знать, где живёт каждый конкретный сервис. API Gateway выступает единой точкой входа: принимает запрос, проверяет авторизацию, терминирует TLS и маршрутизирует запрос в нужный сервис. Без него клиентам пришлось бы знать адреса всех сервисов, а каждый сервис — самостоятельно заниматься аутентификацией и SSL, что быстро превращается в хаос.

### Сравнение решений

| Критерий | Kong | Nginx | Traefik | HAProxy | Envoy (Istio) |
|---|---|---|---|---|---|
| Маршрутизация на основе конфигурации | Да, через Admin API или декларативный конфиг | Да, через конфиг-файлы | Да, автообнаружение + файлы/labels | Да, через конфиг | Да, через xDS API / Istio CRD |
| Аутентификация (JWT, OAuth, API-key) | Да, из коробки через плагины | Базовая (basic auth, JWT через модуль) | Да, через middleware (ForwardAuth, BasicAuth) | Нет из коробки, нужен внешний сервис | Да, через Istio policies |
| Терминация HTTPS | Да | Да | Да, + автоматический Let's Encrypt | Да | Да |
| Управление через UI | Kong Manager (Enterprise), Konga (open source) | Нет (только Nginx Plus Dashboard) | Встроенный веб-интерфейс | Встроенная страница статистики | Kiali (через Istio) |
| Rate limiting | Да, плагин | Да, модуль `limit_req` | Да, middleware | Да, через stick-tables | Да |
| Сервис-дискавери | Да (DNS, Consul) | Нет из коробки | Да (Docker, K8s, Consul, etcd) | Через DNS | Да (K8s нативно) |
| Сложность внедрения | Средняя | Низкая | Низкая | Низкая | Высокая |
| Подходит для Kubernetes | Да (Ingress Controller) | Да (Ingress Controller) | Да (нативная интеграция) | Да (Ingress Controller) | Да (service mesh) |
| Лицензия | Apache 2.0 / Enterprise | BSD (open source) / коммерческая (Plus) | MIT | GPL / коммерческая | Apache 2.0 |

### Мой выбор: Kong

Для крупной компании с микросервисной архитектурой я бы выбрал Kong, и вот почему.

Все три обязательных требования закрываются из коробки без костылей. Маршрутизация настраивается декларативно или через Admin API — удобно и для ручного управления, и для автоматизации через CI/CD. Аутентификация работает через плагины: JWT, OAuth2, API-ключи, LDAP — всё есть и подключается парой строк конфига. TLS-терминация — стандартная функциональность.

Что перевесило в пользу Kong по сравнению с остальными:

Nginx — отличный проверенный инструмент, но для полноценного API Gateway его приходится допиливать. Аутентификация ограничена, нет нормального управления через API, конфиг-файлы при десятках сервисов становятся громоздкими. По сути Kong построен поверх Nginx и решает именно эти проблемы.

Traefik — очень хорош для Kubernetes и Docker, автодискавери — его сильная сторона. Но встроенные возможности аутентификации слабее, чем у Kong. ForwardAuth — это по факту перенаправление на внешний сервис, а не полноценная встроенная проверка.

HAProxy — мощный балансировщик, но не API Gateway. Аутентификации из коробки нет вообще. Использовать его как Gateway — значит строить недостающий функционал самостоятельно.

Envoy/Istio — мощнейшее решение, но избыточно сложное для задачи API Gateway. Istio тащит за собой service mesh со всей его инфраструктурой (sidecar-прокси, control plane). Имеет смысл, если компания уже на Kubernetes и готова к service mesh, но как отправная точка — перебор.

Kong при этом масштабируется горизонтально, имеет большую экосистему плагинов, активное сообщество и возможность перехода на Enterprise-версию, если понадобится поддержка и дополнительный функционал. Для крупной компании с растущей инфраструктурой — оптимальный баланс между возможностями и сложностью внедрения.

## Задача 2: Брокер сообщений

### Сравнение решений

| Критерий | Apache Kafka | RabbitMQ | NATS JetStream | Apache Pulsar | Redis Streams |
|---|---|---|---|---|---|
| Кластеризация | Да, встроенная (KRaft / ZooKeeper) | Да (кворумные очереди) | Да, встроенная | Да (BookKeeper) | Да (Redis Cluster) |
| Хранение сообщений на диске | Да, основной механизм | Да (при настройке persistent) | Да (JetStream) | Да (BookKeeper) | Опционально (AOF/RDB), но ненадёжно |
| Скорость (пропускная способность) | Очень высокая (миллионы msg/s) | Средняя (десятки тысяч msg/s) | Высокая (сотни тысяч msg/s) | Высокая | Высокая, но ограничена RAM |
| Форматы сообщений | Любые (byte array), + Schema Registry | Любые, встроенная поддержка AMQP, MQTT, STOMP | Любые (byte array) | Любые, встроенный Schema Registry | Строки и бинарные данные |
| Разделение прав доступа (ACL) | Да, ACL на топики | Да, vhosts + permissions на очереди/exchanges | Да, accounts + permissions | Да, ACL на топики/namespaces | Да, ACL через Redis ACL |
| Простота эксплуатации | Средняя (стало проще с KRaft) | Высокая | Высокая | Низкая (сложная архитектура) | Высокая, но не брокер в полном смысле |
| Экосистема и комьюнити | Огромная | Очень большая | Растущая | Средняя | Огромная, но не как у брокера |
| Гарантии доставки | At-least-once, exactly-once | At-least-once, at-most-once | At-least-once, exactly-once | At-least-once, exactly-once | At-least-once |
| Retention (хранение истории) | Да, настраиваемый срок хранения | Нет (сообщение удаляется после доставки) | Да, настраиваемый | Да, tiered storage | Ограничено (по длине стрима) |

### Мой выбор: Apache Kafka

Для крупного интернет-магазина с растущей нагрузкой Kafka подходит лучше всего, и вот почему.

По требованиям — всё закрыто. Кластеризация встроена, причём с переходом на KRaft отпала зависимость от ZooKeeper, что упростило эксплуатацию. Сообщения пишутся на диск по умолчанию — это основной принцип работы Kafka, а не опция, которую нужно включать. Пропускная способность — одна из самых высоких среди брокеров, Kafka проектировалась именно под высокие нагрузки. Формат сообщений — любой, а с Schema Registry можно контролировать схемы через Avro, Protobuf или JSON Schema. ACL позволяют гибко разграничивать доступ к топикам на уровне пользователей и групп.

Почему не остальные:

RabbitMQ — отличный брокер для классических очередей с паттерном «отправил — обработал — удалил». Но у нас интернет-магазин, где важно хранить историю событий (заказы, платежи, изменения статусов) и иметь возможность перечитать поток. RabbitMQ этого не умеет — сообщение пропадает после доставки. Плюс по пропускной способности он заметно уступает Kafka.

NATS JetStream — лёгкий и быстрый, отлично подходит для IoT и небольших систем. Но экосистема пока значительно меньше, коннекторов и интеграций с внешними системами не так много. Для крупной компании это риск — можно упереться в ограничения.

Apache Pulsar — по возможностям не уступает Kafka, а в чём-то превосходит (tiered storage, multi-tenancy из коробки). Но архитектура сложнее: зависимость от BookKeeper, больше компонентов в кластере, меньше специалистов на рынке. Для компании, которая только строит микросервисную инфраструктуру, это лишняя сложность на старте.

Redis Streams — это не полноценный брокер, а расширение in-memory базы. Для основной шины событий интернет-магазина полагаться на него рискованно: персистентность ограничена, при переполнении RAM начнутся проблемы.

Kafka при этом де-факто стандарт в индустрии для event-driven архитектур. Огромное количество документации, готовых коннекторов (Kafka Connect), инструментов мониторинга, и найти специалиста с опытом работы с Kafka значительно проще, чем с Pulsar или NATS.
